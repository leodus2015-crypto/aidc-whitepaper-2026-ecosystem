## 2.2 引发思考：AI DC 作为 AI 产业的基础设施，也在发生巨变

> 内容待补充

#### 建议写作要点

- 传统 DC：通用计算、虚拟化、业务系统托管；云 DC：资源池化、弹性服务。
- AI DC 新特征：高算力密度、高网络互联、高数据吞吐、**Token 级成本度量**、**Agent 级工作负载**。
- AI DC 角色变化：从 **资源供应商** 到 **智能能力运营平台**（算力 + 数据 + 模型服务 + Agent Platform）。
- 软硬一体：Facility（电力/散热）— Infra（算力/存储/网络）— Application（Agent 生态）三层联动。

#### 建议核心表述

AI DC 正在从“AI 产业的机房”演进为“AI 产业的 **操作系统 + 工厂**”；其规划对象从机柜与 GPU 数量，扩展为 **瓦特、Token 与 Agent** 三类核心变量。

## 2.3 定义问题：AI 工作负载的不确定性，传统数据中心规划方法论正在失效

> 内容待补充

#### 建议写作要点

- **不确定性来源**：任务步数、Token 消耗、工具调用次数、并发 Agent、上下文长度、模型路由策略均难静态预估。
- **传统方法论失效表现**：按峰值 GPU 采购、按机柜功率粗放规划、按项目烟囱建设 AI、按单次推理计费、缺少 Agent 容量模型。
- **后果**：电力与散热预留不足或过度投资；Token 成本失控；Agent 试点无法规模化；Harness 治理缺失导致生产风险。
- **需要重建的系统化思维**：从单点资源规划走向 **Watt — Token — Agent 协同规划**；从模型中心走向 **Harness + 模型** 整体评估。

#### 建议核心表述

当 AI 工作负载由 Agent 驱动、由 Token 计量、由瓦特约束时，企业必须放弃“按历史业务负载类推”的规划方式，建立面向 **不确定智能任务** 的新容量与成本模型。

## 3.1 以 Watt-centric 构建 AI Facility

> 内容待补充

#### 本节主旨

从 **Watt-centric** 视角规划 AI DC **物理设施层**（Facility）：在 Agentic AI 与高密算力背景下，以 **电力与能效** 为第一性约束，构建可支撑长期演进的供电、散热、空间与能源管理体系。

#### 建议写作要点

- **高功率密度趋势**：AI 机柜从 kW 级走向 tens of kW 甚至更高；GPU / NPU 集群对配电、液冷、气流组织提出新要求。
- **负载形态变化**：训练（持续高功耗）与推理（波动、日内峰谷）并存；**Agent** 带来推理峰谷与 **CPU / 沙箱** 伴生功耗（与 3.2、3.3 衔接）。
- **能效与可持续性**：PUE、液冷、余热回收、绿电与碳排；Watt 不仅是成本，更是 **规模上限**。
- **规划方法**：从“按面积/机柜数”转向“按 **有效瓦特** 与峰值因子”规划；与 Token 容量模型、Agent 并发模型联动（见 2.3、3.2）。
- **企业建议**：预留电力与散热弹性；区分训练区、推理区、通算 / 沙箱区设施策略；建立瓦特—Token—业务 SLA 联动监控。

#### 与 3T 协同

- **Watt → Token**：单位瓦特可支撑的 Token 吞吐（能效比）成为 Facility 与 Infra 的共同 KPI。
- **Watt → Agent**：Agent 峰值并发推理与沙箱扩容直接推高电力与散热需求。

#### 建议结尾

**Watt-centric** 回答 AI DC“物理能不能承载”的问题，是 3T 框架的硬约束层；Facility 规划须与 **3.2 Token-centric Infra**、**3.3 Agent-centric Application** 一体迭代。

## 3.2 以 Token-centric 构建 AI Infra

从 **Token-centric** 视角规划 AI DC **基础设施层**（Infra）：将 **Token** 作为算力消耗、服务成本与容量规划的核心度量单位，构建面向 Agentic 负载的算力、存储、网络与运维体系。

#### 建议写作要点

- **Token 作为中心变量**：输入 / 输出 Token、上下文 Token、缓存命中对成本与性能的影响；企业需建立 Token 预算、配额、分摊与优化机制。

### 3.3 正文示范段（整合版）

> 本节为 3.3 各小节的正文示范段，可作为后续正式写作的语气和观点基准。核心写法从“介绍 Agent 是什么”升级为“阐明企业为什么必须围绕 Agent 构建 AI 生态，以及 AI DC 为什么要承载这一生态”。

#### 3.3.0 正文示范：从使用 AI 工具，到建设 AI 生态

企业进入 Agentic AI 阶段后，真正的命题已经不再是“是否接入一个大模型”或“是否部署几个 Agent”。这些只是 AI 应用的入口动作，而不是企业长期竞争力的来源。我们判断，未来企业竞争的关键，将从“谁使用了更强的模型”转向“谁能围绕自身业务构建一个持续学习、持续反馈、持续优化的 AI 生态”。

企业的核心竞争力一直来自人才、知识、流程和数据的持续积累。人才决定知识创造质量和关键判断能力，流程决定组织能力能否被稳定复用，数据为业务优化提供反馈基础，技术系统则决定这些能力能否被规模化调用。长期以来，企业产出始终受到人力规模约束。一个组织即使拥有优秀人才和成熟流程，要扩大产出，往往仍需要扩大团队、增加管理层级、拉长协同链条。这种“产出与人头线性绑定”的关系，是企业规模化增长的重要天花板。

AI 正在改变这一约束。但企业真正需要的，不是孤立的模型能力，而是一套以 Agent 为入口、以数据和知识为基础、以模型服务为引擎、以流程和评测为反馈、以治理为边界的 AI 生态。它的目标，是把企业的人才经验、组织知识、业务流程和决策规则，转化为可执行、可反馈、可优化、可复制的数字能力。

在这个生态中，Agent 是最重要的执行主体。Agent 不只是辅助人决策的工具，而是能够围绕目标进行理解、规划、执行和交付结果的“数字员工”。但 Agent 的能力不能脱离生态单独存在。没有高质量数据，Agent 无法理解真实业务；没有知识库，Agent 无法沉淀组织记忆；没有流程系统，Agent 无法真正执行任务；没有评测体系，企业无法判断 Agent 是否真的带来业务结果；没有治理机制，Agent 的规模化使用会带来安全、合规和责任风险。

因此，Agent-centric 并不是把 Agent 放在架构图中央，而是承认：企业 AI DC 的 Application 层必须围绕 Agent 这一新型工作负载，重新组织数据、模型、知识、工具、流程、评测和治理能力。工业时代，机器放大体力；信息时代，软件放大流程；AI 时代，Agent 放大人才。但真正决定企业长期竞争力的，不是单个 Agent，而是企业能否构建一个由人、数据、知识、流程、模型和 Agent 共同组成，并能持续自我优化的 AI 生态。

#### 3.3.1 正文示范：企业为什么需要以 Agent 为入口

传统企业软件主要面向人设计。ERP、CRM、OA、MES、ITSM 等系统围绕页面、菜单、表单、流程和权限展开，默认由人理解系统结构、切换系统、拆解任务并逐项执行。换句话说，传统企业的流程运行方式是“人驱动系统操作”。

Agent 带来的变化，不只是多了一个聊天窗口，而是企业软件入口的范式变化。用户不再必须先理解每个系统的操作路径，而是可以直接表达任务目标，由 Agent 负责理解意图、拆解步骤、选择工具、访问数据、调用系统并汇总结果。Agent 的核心范式是“围绕任务组织软件能力”。

这意味着企业运行方式会发生根本变化：人类员工从流程的具体执行者，上移为目标定义者、风险判断者、结果审核者和流程优化者；Agent 则承担大量具体任务的规划、协调和执行。企业各类系统也不再只是被人点击和操作的软件，而是成为 Agent 可以调用的数据层与执行层。Agent 成为连接人、数据、知识、流程和系统的统一智能入口。

以 Agent 为入口的企业，相比传统企业有五类优势。第一，任务闭环更快，从“人找系统”转为“系统随任务聚合”。第二，能力复用更强，岗位 Agent、流程 Agent 和技能包可以在统一平台上复制。第三，知识激活更充分，企业数据和知识不再只被人阅读，而是直接参与任务执行。第四，成本和性能更可运营，任务级 Token、模型路由、工具调用和人类确认都可以被度量。第五，组织弹性更强，企业能力不再完全依赖人力规模，而是可以通过 Agent 复制和数据反馈实现非线性放大。

我们不主张传统界面会短期消失。结构化录入、强监管审批、高风险操作仍需要传统 UI 和人工确认。更现实的趋势是，Agent 成为任务入口和编排中枢，传统软件成为 Agent 可调用的工具、数据源和执行系统。企业需要建设的，不是更多聊天框，而是统一 Agent 工作台和可组合的 Agent 能力体系。

#### 3.3.2 正文示范：企业应该构建什么样的 Agent 平台

如果说 Agent 是企业 AI 生态的执行主体，那么 Agent Platform 就是企业 AI DC Application 层的操作系统。企业需要的不是一个可以快速做 Demo 的 Agent Framework，而是一个能够长期运行、可审计、可评测、可恢复、可治理的生产级 Agent Platform。

这个平台首先要支撑 Agent 的分层体系。企业至少会形成三类核心 Agent：个人 Agent、岗位 Agent 和组织 Agent。个人 Agent 服务每个员工，管理时间、知识、任务、会议、沟通和个人工作流；岗位 Agent 服务销售、研发、客服、HR、采购、财务等具体岗位，沉淀岗位知识、业务规则和流程经验；组织 Agent 服务企业整体，跨部门、跨系统、跨流程调度资源，围绕业务目标协调多个岗位 Agent 共同完成任务。

每类 Agent 都必须有“工作手册”。工作手册类似数字员工的岗位说明书和操作规程，需要明确 Agent 的职责范围、上下级关系、可调用工具、可访问数据、权限边界、人工确认条件、异常升级路径和审计要求。没有工作手册的 Agent，不应进入生产流程；没有权限边界的 Agent，不应调用企业系统；没有评测与回放的 Agent，不应承接关键任务。

从架构上看，生产级 Agent Platform 至少包括六层能力：入口层承接用户目标和业务场景；编排层负责任务规划、多 Agent 协同、队列与恢复；运行层负责 Agent Runtime、状态持久化、上下文工程和模型调用；能力层连接模型服务、数据服务、工具和企业系统；安全层提供沙箱、权限、策略和审计；运营层提供 AgentOps、评测、监控、Token 成本和任务回放。

这套平台的本质，是围绕模型构建 Harness 控制平面。Agent 的可靠性越来越不只取决于模型本身，而取决于模型是否被放进一个可执行、可观测、可验证、可治理的系统中。执行环境、工具接口、上下文管理、生命周期编排、可观测性、验证体系和治理机制，构成 Agent 进入生产环境的基本条件。

企业落地路径也应循序渐进。第一阶段是“员工管理 Agent”，由员工授权、监督、确认和纠偏；第二阶段是“Agent 协助管理流程”，由流程 Agent 分派任务、汇总结果，但关键节点保留人工确认；第三阶段才是受控的“Agent 管理 Agent”。未来一年，企业建设重点不应是追求 Agent 数量，而是优先补齐 Tool Registry、Sandbox Pool、Durable Workspace、Trace、任务级 Token 账单、Handoff Contract 和细粒度权限。

#### 3.3.3 正文示范：Data for AI，不是把文档扔进向量库

企业的数据资产要服务 Agent，必须从 Data for BI 升级为 Data for AI。传统数据平台主要服务报表、分析和经营决策，核心对象是人；Agent 时代，数据要直接参与推理、检索、引用、记忆和任务执行，核心对象变成了 Agent。

因此，Data for AI 的目标不是“把文档扔进向量库”，而是让数据以 Agent 可调用、可治理、可担责的方式参与任务执行。企业数据需要从“可存储、可分析”升级为“可检索、可引用、可授权、可追溯、可更新”。只有做到这一点，Agent 才能在真实业务环境中使用企业专属知识，而不是只依赖通用模型的泛化能力。

Data for AI 至少包括五项改造。第一，非结构化知识资产化。制度、手册、合同、工单、日志、代码、会议纪要等，需要经过解析、切片、向量化、索引和权限绑定，成为 Agent 可调用的知识资产。第二，RAG 升级为知识服务。企业需要的不只是“能搜到”，而是混合检索、重排、引用溯源、拒答策略和知识刷新 SLA。第三，Agent Memory 分级治理。工作记忆、岗位记忆和组织记忆应有不同保留周期、审批流程和过期策略。第四，状态估计化。进入 Agent 上下文的信息，应携带来源、时间、有效期、置信度和是否可自动采纳。第五，实时数据管道化。订单、告警、审批、工单等业务事件，应通过流式接口进入 Agent 上下文，缩短感知到行动的延迟。

我们判断，未来 Agent 失败的相当一部分，不会来自模型不够聪明，而会来自数据过期、引用错误、权限继承不完整或上下文腐烂。企业如果只建设向量库，而不建设数据血缘、知识刷新、权限继承和引用溯源，就会得到一个看似智能、实则不可担责的 Agent。

因此，Data for AI 的验收标准应是四句话：用得上，Agent 能稳定检索和引用正确知识；管得住，权限、血缘、脱敏和合规可执行；信得过，知识过期和引用错误可发现、可追责；更得起，知识更新有 SLA，而不是上线即腐烂。

#### 3.3.4 正文示范：模型服务化成为主流

Agent 时代，企业不应再把模型看成单个产品，而应把模型看成一组可运营的服务。模型生态的成熟标志，不是企业接入了多少大模型，而是能否把多模型能力转化为稳定、可治理、可计量、可按任务调用的模型服务。

模型服务化的核心，是通过 Model Gateway、Model Router、Inference Server、Model Registry、Evaluation Platform 和 ModelOps，把通用模型、行业模型、企业专属模型、小模型、多模态模型统一纳入服务层。业务应用和 Agent 不直接连接某个模型，而是通过模型服务层获得推理能力。这样，企业才能统一鉴权、协议、计量、限流、审计、版本、灰度和回滚。

Agent 使模型服务化变得更加必要。一个 Agent 任务往往包含规划、检索、判断、执行、校验、总结等多个步骤，并不需要每一步都调用最强模型。复杂规划可以使用高能力模型，结构化抽取可以使用小模型，敏感数据任务可以使用私有化模型，多模态任务可以调用专用模型。Model Router 的价值，就是根据任务类型、步骤位置、剩余预算、延迟 SLA、数据敏感级和模型表现，动态选择最合适的模型服务。

因此，企业模型策略应从“选一个最强模型”转向“为 Agent 每一步选最合适模型”。这会带来三个直接收益：降低单位任务 Token 成本，提高响应速度，降低敏感数据暴露风险。同时，模型评测也必须从通用 Benchmark 转向企业任务轨迹。模型升级不应只看榜单分数，而应触发 Harness 级回归测试，验证对真实 Agent 任务链路的影响。

未来一年，模型服务化的成熟度可以用三个信号衡量：第一，所有 Agent 模型调用都经过 Gateway，不允许影子直连；第二，任务级 Token 账单可以下钻到模型 SKU 和任务步骤；第三，模型变更能够自动触发回归评测和灰度发布。谁能做到这些，谁就能把模型从技术资产变成可运营的生产能力。

#### 3.3.5 正文示范：应用生态是 Agent 能力的业务出口

AI DC 的最终价值，必须通过业务应用释放出来。Agent-centric 的应用生态，不是让每个系统旁边多一个聊天窗口，而是让 Agent 进入岗位、流程和行业场景，把企业的知识、数据、模型和工具能力嵌入真实业务链路。

企业 AI 应用会经历三个阶段。第一阶段是个人 Copilot，主要提升写作、检索、总结、分析等个人效率。第二阶段是岗位 Agent，将销售、客服、研发、运维、财务、人力等岗位的知识、规则、案例和操作流程沉淀为可复用能力。第三阶段是流程 Agent 和组织 Agent，跨系统、跨部门、跨步骤执行复杂任务，例如商机跟进、采购审批、故障处理、财务对账和经营分析。

在这个过程中，Agent 的价值从“辅助建议”推进到“可审计的半自动执行”。完全无人值守只适合低风险、可回滚场景；大多数企业流程更现实的模式，是 Agent 完成信息收集、方案生成、填单、调用系统、初步校验，人类负责关键判断、风险确认和责任承担。

应用生态能否规模化，取决于是否依托统一 Agent Platform。如果每个部门各自建设 Copilot、各自接模型、各自封装工具，企业最终会得到一批难以治理、难以复用、难以计费的影子 Agent。反之，如果岗位 Agent、流程 Agent 和行业能力包都运行在统一平台上，企业就可以复用模型服务、数据服务、工具目录、权限策略、评测体系和 Token 预算。

因此，应用落地的优先级不应从“最炫酷场景”开始，而应从高频、可评测、可回滚、有明确 ROI 的流程开始。建议优先顺序是：高频可评测流程，知识密集型岗位，跨系统长流程，最后才是探索型创意场景。Agent 应用的成功，不是一个 Demo 的惊艳，而是能否在统一平台上被复制、被度量、被治理。

#### 3.3.6 正文示范：治理不是刹车，而是规模化前提

Agent 进入企业生产系统后，治理不再是合规部门的事后检查，而是 AI DC Application 层的内建能力。Agent 可以访问数据、调用工具、操作系统、生成内容、触发流程，因此企业必须回答清楚：谁授权 Agent 做了什么，Agent 基于哪些数据和模型做出动作，过程是否可回放，结果是否可验证，责任由谁承担。

治理的核心包括三类能力。第一是身份、权限与访问控制。Agent 应拥有独立数字身份，权限应细化到数据、工具、模型和任务，并能随任务生命周期即时收回。第二是观测、审计与追溯。企业需要记录模型调用、检索片段、工具入参和出参、人工确认点、Token 成本和最终动作，支持按业务单据追溯。第三是验证与评测。生产失败轨迹应自动进入回归集，模型变更、工具变更和 Harness 变更都要经过联合评测。

Agent 治理还必须与成本治理结合。过去企业看 AI 成本，往往只看模型调用账单；Agent 时代，成本来自 Token、模型路由、工具调用、沙箱运行、检索、Trace 存储和人工确认。企业应建立组织、部门、任务类型三级 Token 预算，并把 Watt 峰值、Token 峰值和 Agent 并发放在同一张运营视图中。

我们不建议企业直接追求“全自主 Agent”。更现实的一年目标，是从 L2 治理成熟度走向 L3：有工作手册、有 Token 账单、有抽样评测，进一步升级为 Trace-native 回归、细粒度权限、标准化人工交接和任务级成本闭环。治理不是限制 Agent，而是让 Agent 能够进入关键业务系统的前提。

#### 3.3.7 正文示范：学习闭环才是企业 AI 生态的护城河

企业构建 AI 生态的核心，不是简单自动化，而是形成学习闭环。一个成熟的企业 AI 生态，应形成这样的循环：人类定义目标和标准，Agent 执行业务任务，系统记录执行过程和结果，数据反馈进入知识库和评测体系，企业判断哪些结果有效、哪些流程需要优化，Agent 在反馈中持续改进，新一轮任务产生更高质量的数据和经验。

这个闭环会让企业的知识、流程和判断力不断复利。每一次任务执行，都会产生新的业务数据；每一次人类反馈，都会沉淀新的判断标准；每一次流程优化，都会提升 Agent 下一次执行质量。久而久之，企业会形成一套带有自身业务理解、组织记忆和行业经验的 AI 系统。这才是 AI 时代企业真正的新 IP。

企业未来需要同时积累两类资本：人力资本和 AI 资本。人力资本包括人的知识、经验、判断力、创造力、客户关系和组织理解；AI 资本则包括企业专属 Agent、私有知识库、业务数据资产、工作流数据、私有评测体系、模型编排能力、任务执行轨迹、人类反馈数据以及安全治理体系。AI 资本的价值，在于把人力资本中的经验、判断和方法沉淀到系统中，并通过 Agent 进行规模化调用。

这不意味着人力资本不重要。相反，AI 资本越强，人力资本越重要。因为没有人类设定目标、校准方向、判断风险和设计流程，AI 只是在消耗算力。真正有价值的 AI 系统，不是自动运行的黑箱，而是在人类目标牵引下不断学习和优化的组织能力系统。

因此，企业能力的增长逻辑正在从“企业能力 = 人才数量 × 人均效率”，逐步演变为“企业能力 = 人力资本 × AI 资本 × 数据质量 × 算力供给 × 治理能力”。一个企业是否真正掌握 AI 主权，关键测试是：能否替换通用模型，同时保留自身 AI 生态中沉淀下来的“公司老员工式经验”。谁能率先构建这一学习闭环，谁就能在 AI 时代形成新的组织智能和竞争壁垒。

### 3.3.1 以 Agent 为入口的 AI DC 生态

#### 本节主旨

说明 AI DC 不仅是算力集群，更是 **以 Agent 为入口**、连接数据、模型、应用与传统企业软件的生态平台。重点阐明「**入口 — 能力组件 — 3T 底座**」关系。

#### 写作逻辑

1. 传统 DC / 云 DC：人通过界面操作软件。
2. Agent 时代：用户 **以 Agent 为入口** 表达目标，触发长任务、多轮推理、工具调用、RAG 与多 Agent 协同。
3. 企业需要可被 Agent 调用的模型**服务**、数据**服务**、工具与企业系统能力。
4. 生态越丰富，越需要统一治理；Token 成本见 **3.2**，电力约束见 **3.1**。

#### 关键观点

- AI DC 的首要生态指标之一，是能否作为 **Agent 统一入口** 稳定承载任务及伴随的 Token 与数据访问模式。
- 算力、存储、网络、数据、模型、Agent 与应用深度耦合，单点优化难以支撑规模化。
- 企业应以平台工程思路建设 AI DC，使各类 Agent 与智能应用复用同一套入口、Harness 与生态能力。
- **一年展望**：入口形态从“多个部门级 Copilot”收敛为 **企业级 Agent 工作台**；传统 GUI 与 Agent 入口 **长期并存**，而非一夜替代。

#### 建议展开段落

##### 3.3.1.1 从资源供给到以 Agent 为入口的能力供给

说明企业用户日益 **以 Agent 为入口** 表达业务目标；AI DC 需交付 **Agent Harness**、数据检索、**模型服务**、工具集成与治理能力，而非仅交付裸算力或单次模型 API。未来一年，能力供给的标准交付件应从“模型 API 密钥”升级为 **可编排的 Agent 能力包**（含工具、权限、记忆边界与成本配额）。

##### 3.3.1.2 Agent 入口驱动 AI-Native 平台架构

训练、推理、RAG、Agent 执行、沙箱与编排共同构成 AI-Native 平台；Agent 入口放大 Token 消耗，容量模型在 **3.2** 量化；电力峰值在 **3.1** 预留。**2027 方向**：企业侧 **推理与 Agent 执行占比持续上升**，入口流量峰谷更明显，须在 3T 容量模型中单独列出 **Agent 并发与后台常驻 Agent（ambient / scheduled）** 场景。

##### 3.3.1.3 以入口为中心的生态治理

数据、模型、Agent、工具、应用共享同一底座时，须统一治理，避免烟囱式 AI 应用。一年内，治理重点从“能否调用大模型”转向 **“同一入口下的工具、数据、模型、预算是否同一套策略”**。

##### 3.3.1.4 统一入口与多形态共存（一年展望）

- **统一 Agent 工作台**：聚合员工 / 岗位 / 流程 Agent，提供任务历史、权限视图与人工交接入口。
- **Ambient Agent**：后台监控、周期报表、异常巡检类 Agent 与交互式入口并存，对 **O 层观测与 Token 基线负载** 提出新要求。
- **复合交互**：高结构化录入、强监管审批等场景仍保留传统 UI；Agent 负责 **编排与填单**，人负责 **确认与担责**。

#### 建议结尾

AI DC 建设目标应从“拥有算力”迈向“**以 Agent 为入口运营智能能力**”；未来一年，入口建设的关键不是再做更多聊天框，而是 **统一工作台 + 可组合 Agent 能力**；Harness 与平台见 **3.3.2**。

### 3.3.2 构建 Agent 时代的数据中心底座

#### 本节主旨

**3.3 章节重心。** 论述 **Agent-centric** 软件底座：以 Agent 为入口与核心工作负载，通过 **Harness（ETCLOVG）**、**Agent Platform** 及与 **3.1 Watt / 3.2 Token** 的协同，形成可规模化落地的 Application 层底座。本节侧重 **软平台**；硬 Infra 见 3.2，Facility 见 3.1。

#### 写作逻辑

1. Agent 入口范式 vs 传统人机界面（3.3.2.1）。
2. 企业 Agent 分层、工作手册与管理演进。
3. Harness Engineering 与 ETCLOVG。
4. Agent Platform 参考架构。
5. 对 3.1 / 3.2 的关键诉求摘要（3.3.2.5）。

#### 关键观点

- **Agent 时代数据中心底座** = **3.1 Facility + 3.2 Infra + 本节 Harness / Agent Platform**。
- **Binding-constraint**：长周期可靠性日益取决于 harness 与模型共同作用；一年内业界将更多承认 **“换模型解决不了的生产失败”** 实为 Harness 缺口。
- 生态从 **Agent Framework** 走向 **Agent Platform**；2027 年企业采购与自建的重点是可运营 Platform，而非单次 Framework 集成。
- Token 与算力规划须与 Agent 容量模型对齐（**3.2**）。
- **一年展望**：企业 IT 预算从“模型许可”逐步倾斜至 **Agent Platform、工具封装、观测与治理**；Harness 的 **O / G 层** 由短板变为必选项。

#### 建议展开段落

##### 3.3.2.1 从面向人的企业软件，到面向任务的 Agent 入口

说明传统企业软件主要是面向人设计的。ERP、CRM、OA、MES、ITSM 等系统通常围绕页面、菜单、表单、流程和权限展开，人需要理解系统结构，在多个系统之间切换，把业务目标拆解为一系列操作步骤，再逐项完成执行。传统软件的核心范式是“人使用软件”。

Agent 则代表一种新的软件入口范式。用户不再必须先理解每个系统的操作路径，而是可以直接表达任务目标，由 Agent 负责理解意图、拆解步骤、选择工具、访问数据、调用系统并汇总结果。Agent 的核心范式是“围绕任务组织软件能力”。

在 AI Native 企业中，Agent 将成为越来越多任务型、流程型和知识型场景的新入口。传统企业软件不会因此消失，而会逐步成为 Agent 可调用的工具、数据源和执行系统。Agent 位于用户目标与企业系统之间，把分散在不同软件中的能力重新编排起来，从而推动企业流程从“人驱动系统操作”走向“任务驱动智能协同”。

Agent 对企业软件的重塑不是简单增加聊天窗口，而是改变交互方式与流程组织。须为这一入口提供 Agent Runtime、工具注册、权限控制、沙箱隔离、任务审计与 Harness 治理；Token 与电力诉求分别见 **3.2**、**3.1**。

**一年展望（2027）**：传统企业软件供应商将更多提供 **Agent-ready API / 事件 / 工具描述**；企业内 **“系统即工具”** 封装项目会成为 Agent 落地的前置工程，与 ERP / OA 存量改造并行，而非等待全新替换。

##### 3.3.2.2 企业 Agent 会形成分层体系

企业中的 Agent 不会以单一形态存在，而会围绕岗位、流程、专业能力和行业场景形成分层体系。可以区分四类 Agent：

- 员工 Agent：作为个人助理，帮助知识工作者完成检索、写作、分析、会议、代码和办公任务。
- 岗位 Agent：面向销售、客服、研发、财务、人力、运维等岗位，沉淀岗位知识和操作流程。
- 流程 Agent：跨系统、跨部门、跨步骤执行复杂业务流程，例如商机跟进、采购审批、故障处理和报告生成。
- 行业 Agent：结合行业知识、专有数据和业务规则，服务制造、金融、政务、能源、医疗等场景。

随着 Agent 深入企业流程，企业需要为每类 Agent 定义清晰的“工作手册”。这类工作手册类似数字员工的岗位说明书和操作规程，需要明确 Agent 的职责范围、上下级关系、可调用工具、可访问数据、权限边界、人工确认条件、异常升级路径和审计要求。

在落地路径上，企业宜先采用“员工管理 Agent”的方式，由员工对 Agent 进行授权、监督、确认和纠偏。随着评测、审计、权限和运行治理能力成熟，再逐步过渡到“Agent 管理 Agent”的多智能体协作模式。即便进入多 Agent 协作阶段，高风险操作、关键业务决策和敏感数据访问仍应保留人工确认和审计机制。

Agent 的普及还会改变企业员工的岗位职责和工作流程。员工将从具体系统操作者，逐步转变为目标定义者、Agent 管理者、结果审核者和流程优化者。企业组织也可能围绕 Agent 形成人机混合团队、Agent 主管、数字员工运营、AgentOps 管理等新角色，从而推动企业组织架构向 AI Native 形态演进。

**一年展望（2027）**：

- **能力包（Skills）化**：岗位 Agent 由“一个大 Prompt”拆为可版本化、可评测的 **技能模块**，便于复用与权限细分。
- **Supervisor–Worker 常态化**：流程 Agent 作为主管分派子任务给岗位 Agent，工作手册须定义 **委托边界与升级条件**。
- **数字员工身份**：Agent 纳入 IAM（身份、角色、审计主体），与员工账号 **平行但不可混淆**。
- **组织 KPI 变化**：从“人均产出”扩展为 **“人机团队任务完成率、单位任务 Token 成本、人工确认率”**。

##### 3.3.2.3 驾驭工程（Harness Engineering）与 ETCLOVG 控制平面

**参考**：*Agent Harness Engineering: A Survey*（OpenReview, 2026）；Martin Fowler、OpenAI 等业界实践。

###### 核心判断（binding-constraint thesis）

在长周期、工具调用、多步骤任务中，Agent 的现实可靠性日益取决于 **agent execution harness**，而不只是底层模型。对企业而言，**构建 Agent 时代数据中心底座的核心，是建设可信赖的 Harness 控制平面**。

| 维度 | 传统视角 | Harness 视角 |
| ---- | -------- | ------------ |
| 可靠性来源 | 主要由模型能力决定 | 模型 + harness 共同决定，harness 可能成为绑定约束 |
| 工程重点 | Prompt、少量工具、单轮交互 | 沙箱、工具协议、上下文、编排、观测、验证、治理的闭环 |
| 评估对象 | 模型分数 | 模型 + controller + runtime + feedback 的整体行为 |
| 失败诊断 | “模型不够聪明” | 工具 schema、沙箱、上下文漂移、测试不稳、judge、权限等 |

###### 三阶段演进：Prompt → Context → Harness

| 阶段 | 核心对象 | 典型机制 | 局限 |
| ---- | -------- | -------- | ---- |
| Prompt Engineering | 单次输入文本 | 指令、示例、格式约束 | 难处理长周期状态、工具反馈与执行失败 |
| Context Engineering | 每一步可见信息 | 检索、摘要、长期记忆、缓存、上下文压缩 | 未解决权限、沙箱、观测、验证与治理 |
| **Harness Engineering** | **闭环控制系统** | 执行环境、工具协议、编排、观测、验证、治理 | 成本与跨层耦合显著提高 |

###### ETCLOVG 七层分类

| 层级 | 含义 | 与 3T / 章节映射 |
| ---- | ---- | ---------------- |
| **E** Execution Environment | 执行环境与沙箱 | **3.2** 沙箱 / CPU；**3.1** 伴生功耗 |
| **T** Tool Interface | 工具接口与协议 | Tool Registry |
| **C** Context Management | 上下文与记忆 | **3.3.3** 数据生态 |
| **L** Lifecycle / Orchestration | 生命周期与编排 | 编排层、工作手册 |
| **O** Observability | 可观测性与运维 | AgentOps；**第四章** 运营实践 |
| **V** Verification | 验证与评估 | 评测平台、trace-native eval |
| **G** Governance | 治理与安全 | **3.3.6** |

**写作提示**：开源生态在 E、T、L、V 较密集；**O 与 G 相对薄弱**，是企业应重点补齐的能力。Harness 局部改动可能产生 **跨层耦合**，应按系统变更测试。

###### 一年演进重点（2026 → 2027，按 ETCLOVG）

| 层级 | 未来一年企业应优先补强 |
| ---- | ---------------------- |
| **E** | 生产级沙箱默认开启；代码 / 浏览器 / 桌面自动化 **隔离与复位** 标准化 |
| **T** | 工具 **Schema + 风险分级 + 协议标准**（企业内 Tool Registry 与 MCP 类对接并存） |
| **C** | 上下文 **压缩 + 过期策略 + 来源标注**；避免 context rot 成为主要失败模式 |
| **L** | 长任务 **断点续跑、超时与重试策略**；多 Agent 交接协议产品化 |
| **O** | **全链路 trace + Token 细账 + 任务回放** 进入 AgentOps 标配 |
| **V** | 生产轨迹 **回归测试集**；按业务任务而非通用榜单验收 |
| **G** | **按工具 / 数据 / 模型的细粒度授权**；高风险操作 **强制人工批准** |

###### 第四阶段预告（提纲级，勿写死）

在 Prompt → Context → Harness 之后，业界可能出现 **Evaluation / Operations Engineering** 独立成势：即 **“先能运营、再扩能力”**。企业一年内宜把 **可观测、可回归、可担责** 作为 Agent 扩面的前置条件，而非事后补课。

###### 三组系统级权衡

| 权衡 | 工程含义 |
| ---- | -------- |
| 成本—质量—速度 | 检查分层：同步 / 异步 / 回归 / 人工升级（衔接 **3.2 Token**、**3.1 Watt**） |
| 能力—控制 | 工具、权限、身份、审计、human approval 一体设计 |
| 跨层耦合 | 评测结果应反馈编排与恢复策略 |

###### 从 Framework 到 Platform

| 能力 | Framework | **Agent Platform** |
| ---- | --------- | ------------------ |
| 运行 | agent loop、tools、memory | 持久工作区、托管沙箱、可恢复任务 |
| 运营 | 基础日志 | 观测、评测、治理、标准化人工交接 |
| 演进 | 静态堆叠 | 随模型能力升级 **动态简化 harness** |

**建议核心表述**：Agent-centric 底座 = **3.1 Facility + 3.2 Infra + Harness / Agent Platform（ETCLOVG）**。

##### 3.3.2.4 Agent 平台参考架构建议（映射 ETCLOVG）

在企业级场景中，应沉淀为统一的 **Agent Platform**（见 3.3.2.3），向上支撑四类 Agent，向下连接模型服务、数据服务、工具、沙箱与企业系统。

**与 ETCLOVG 的映射**：

- **E**：托管沙箱；与 **3.2** 统一供给。
- **T**：Tool Registry、工具 schema、风险分级、企业系统封装。
- **C**：Runtime 上下文编排 + **3.3.3** 检索 / 记忆 / 压缩。
- **L**：多 Agent 编排、工作手册、人工升级。
- **O**：全链路 trace、**Token** 计量、任务回放。
- **V**：轨迹评测、业务回归、红队。
- **G**：身份、权限、审计；与 **3.3.6** 统一。

**平台分层**：入口层 → 编排层（L）→ 运行层（C + Runtime）→ 能力层（T + 模型 / 数据）→ 安全层（E + G）→ 运营层（O + V）。

**一年展望（2027）— Platform 必备能力清单**：

- **Durable Workspace**：任务级持久化工作区（文件、分支、中间产物），支持中断恢复。
- **Managed Sandbox Pool**：沙箱池化调度，与 **3.2** CPU / 内存容量联动。
- **Tool & Model Registry 统一目录**：Agent 仅通过目录发现能力，禁止“影子 API”。
- **Handoff Contract**：人工交接的结构化载荷（意图、权限、预算、风险、未决项）。
- **Per-task FinOps**：单任务 Token / 工具调用 / 外部 API 成本上限与告警。

##### 3.3.2.5 Agent 对 3.1 Facility / 3.2 Infra 的关键诉求（衔接 3T）

Agent 显著改变资源与成本结构；核心度量之一是 **Token**，并伴生 **Watt** 与 **CPU** 压力。

| 诉求维度 | Agent 带来的变化 | 规划归属 |
| -------- | ---------------- | -------- |
| 推理算力 / Token | 多轮推理；Token 可能百倍级增长 | **3.2** |
| 通用计算 | 编排、沙箱、工具调用、网关、审计 | **3.2** |
| 上下文与记忆 | 超长上下文、长记忆、状态持久化 | **3.2** / **3.3.3** |
| 存储与缓存 | KV Cache、向量库、trace 存储 | **3.2** |
| 电力与散热 | 推理峰谷、高密机柜、沙箱伴生功耗 | **3.1** |
| 容量模型 | 任务步数、每步 Token、并发 Agent | **3.2**（3T 联动） |
| 常驻 / 后台 Agent | 巡检、监控、周期任务形成 **Token 与 Watt 基线负载** | **3.2** / **3.1** |
| Trace / 评测存储 | 生产轨迹留存与回归集构建 | **3.2** |

**一年展望**：Agent 负载模型须新增 **后台 Agent 占比、峰值并发系数、工具调用放大因子** 三参数；企业 Infra 规划若仍只按“对话用户数”估算，将在一年内出现 **Token 与电力双重低估**。

**写作提示**：本节保持“问题清单 + 指向 3.1 / 3.2”，避免重复硬件细节。

#### 对 Agent-centric 底座的要求（汇总）

- **3.1**：有效瓦特、散热弹性、峰谷电力策略。
- **3.2**：Token 计量、推理算力、CPU / 沙箱、KV Cache、trace 存储。
- **ETCLOVG**：E / T / C / L / O / V / G 全链路（见上）。

#### 建议结尾

**构建 Agent 时代的数据中心底座**，要求 **3T 一体**：**Watt（3.1）+ Token（3.2）+ Harness（3.3.2）**。未来一年，底座竞争焦点将从 **“能否跑通 Agent”** 转向 **“能否稳定运营一万个任务长周期 Agent”** —— 取决于 Platform 与 O/G/V 是否到位。

### 3.3.3 数据生态：支撑 Agent 的 AI-Ready Data

#### 本节主旨

以 **Agent 消费数据** 为主线，说明数据生态是 Harness **C 层** 能力：为 Agent 提供可检索、可引用、可记忆、可治理的企业知识与业务数据。

#### 写作逻辑

1. 传统数据平台面向 BI；Agent 要求数据直接参与推理与执行。
2. Harness C 层：检索、摘要、记忆、缓存、压缩；走向 **state estimation**（来源、时效、不确定性）。
3. RAG、向量库、知识图谱与 Agent Memory。
4. 数据治理是 G 层前置条件。
5. **数据服务化**：API / 检索服务供 Agent 与模型服务调用。

#### 关键观点

- 没有 AI-Ready Data，Agent 入口难以形成企业专属智能。
- 持久记忆越强，越需 provenance 与过期治理。
- 权限、血缘、质量与合规决定 Agent 能否进入生产流程。
- **一年展望**：RAG 从“能检索”升级为 **“带 SLA 的知识服务”**；Agent 失败越来越多源于 **数据过期与引用错误**，而非模型能力不足。

#### 建议展开段落

##### 3.3.3.1 从数据可存储到 Agent 可使用

数据须以 **服务** 形态供 Agent 调用（检索、订阅、事件推送），并声明 **时效、覆盖范围与权限域**。

##### 3.3.3.2 非结构化数据成为 Agent 知识资产

文档、工单、日志、代码等需 **管道化入湖 / 入向量库**；一年内企业会更多建设 **面向 Agent 的知识切片规范**（chunk 策略、元数据、版本）。

##### 3.3.3.3 向量库与知识库支撑 Agent 检索增强

混合检索（向量 + 关键词 + 重排）成为默认；**引用溯源** 与 **拒答策略** 进入生产验收项。

##### 3.3.3.4 知识图谱补充 Agent 业务语义

复杂流程场景下，图谱提供 **规则、关系与约束**；与向量检索互补，降低“相似但错误”的召回。

##### 3.3.3.5 数据治理保障 Agent 可信运行

数据权限须 **继承到 Agent 会话与工具调用**；一年内 **行级 / 文档级 / 工具级** 统一策略成为合规硬要求。

##### 3.3.3.6 记忆与上下文的一年演进（2027）

- **Agent Memory 分级**：工作记忆 / 岗位记忆 / 组织记忆，不同保留周期与审批流程。
- **State estimation**：上下文条目携带 **来源、时间、置信度、是否可自动采纳**。
- **实时数据接入**：业务事件（订单、告警、审批）通过流式接口进入 Agent 上下文，缩短 **感知延迟**。

#### 建议结尾

数据生态让 **Agent 入口** 的任务链路“用得上、管得住、信得过”企业数据资产；未来一年，**知识刷新 SLA 与记忆治理** 将与模型选型同等重要。

---

### 3.3.4 模型生态：模型服务化成为主流

#### 本节主旨

说明企业模型建设走向 **模型服务化成为主流**：通过 Model Gateway、推理服务、Model Router、Model Registry 与 ModelOps，将多模型能力封装为 Agent 可调用的 **标准模型服务**，在能力、**Token 成本**、延迟与安全间平衡，作为 Harness 内 **推理引擎**。

#### 写作逻辑

1. 服务化趋势：统一模型服务层。
2. 多模型并存与分层。
3. Gateway、Router、Inference Server、评测、ModelOps。
4. Agent 场景选型与 **3.2** Token 优化。
5. V 层评测须在 harness 语境下解读。

#### 关键观点

- **模型服务化成为主流**：统一经 Gateway 获取推理能力，**Token 可计量**。
- 模型是 harness 内推理引擎，不宜脱离 runtime 单独评估。
- Model Router 控制 Agent 链路 Token 成本与延迟。
- **一年展望**：企业模型策略从 **“选一个最强模型”** 转向 **“为 Agent 每一步选最合适模型”**；推理服务成为 AI DC 的 **日常运营核心**，而非训练一次性工程。

#### 建议展开段落

##### 3.3.4.1 模型服务化：从模型资产到可调用的生产能力

统一 **鉴权、协议、计量、限流、审计、版本**；对外只暴露 **模型服务 SKU**（能力标签 + 单价 Token + 上下文上限 + 数据驻留策略）。

##### 3.3.4.2 多模型并存与服务分层

**规划 / 推理 / 执行 / 抽取 / 多模态** 分层；一年内 **小模型执行 + 大模型规划** 的级联调用成为 Agent 降本默认模式。

##### 3.3.4.3 Model Gateway 与 Model Router

Router 输入除任务类型外，应纳入 **步骤位置、剩余预算、延迟 SLA、数据敏感级**；与 **3.3.2** Harness 编排联动，避免 Router 与 Agent 编排 **双重决策冲突**。

##### 3.3.4.4 推理服务决定服务化生产体验

Agent 场景重视 **首 Token 延迟、长上下文 KV Cache 命中、批处理与并发隔离**；一年内 **Agent 专用推理池**（与交互式聊天池分离）将成为常见架构选择。

##### 3.3.4.5 面向 Agent 的模型评测与持续回归

评测集须为 **企业任务轨迹**；模型升级触发 **Harness 级回归**（非仅模型榜单对比）。固定 harness 下对比模型，方可体现 binding-constraint。

##### 3.3.4.6 一年展望：模型服务化的三个成熟度信号（2027）

1. **100% Agent 调用经 Gateway**，无影子直连。  
2. **任务级 Token 账单** 可下钻到模型 SKU 与步骤。  
3. **模型变更 → 自动回归 → 灰度发布** 成为标准变更流程。

#### 建议结尾

模型生态成熟的标志，是 **模型服务化** 能否支撑 **Agent 入口** 的生产任务——稳定、可治理、**Token 可运营**（衔接 **3.2**）；一年内，**Router + 分层模型 + 轨迹评测** 将比追加单一超大模型更能改善 Agent 生产表现。

---

### 3.3.5 应用生态：Agent 能力的业务出口

#### 本节主旨

应用生态是 **Agent-centric** 价值的最终释放层：从 Copilot 到岗位 / 流程 / 行业智能应用，强调 **嵌入流程** 而非孤立聊天窗口。

#### 写作逻辑

1. 早期 AI 应用与 Agent 化应用差异。
2. 岗位 Agent、流程 Agent、行业应用、低代码 / 插件。
3. 与 ERP、CRM、MES、OA 等集成。
4. 应用级治理与 ROI（**第四章** 实践衔接）。

#### 关键观点

- 应用繁荣依赖 3.3.2 统一底座与 3.3.4 模型服务，避免重复建设。
- 流程 Agent 把 AI 从“建议”推进到“协同执行”。
- **一年展望**：企业 AI 应用从 **“辅助建议”** 为主，转向 **“可审计的半自动执行”** 为主；完全无人值守仍限于低风险、可回滚场景。

#### 建议展开段落

##### 3.3.5.1 从 Copilot 到企业智能应用

Copilot 仍扩面，但价值重心转向 **可重复、可评测、可回滚的任务自动化**；一年内 **岗位 + 流程 Agent 组合** 将取代大量单点 Copilot 集成。

##### 3.3.5.2 岗位 Agent 沉淀专业能力

岗位 Agent 以 **技能包 + 工作手册 + 业务指标** 三件套交付；与 HR / 岗位体系对齐，便于权限与审计。

##### 3.3.5.3 流程 Agent 连接企业系统

流程 Agent 须对接 **事件驱动架构**（审批完成、告警触发）；工具封装优先于 UI 自动化，降低脆弱性。

##### 3.3.5.4 行业智能应用形成差异化价值

行业 Agent 结合 **行业数据服务 + 合规规则 + 行业工具链**；一年内更多以 **“行业能力包”** 形式在 AI DC 内发布，而非每个项目定制。

##### 3.3.5.5 应用生态需要统一集成与治理

应用上线须绑定 **Token 预算、权限模板、trace 留存周期**；低代码 / 无代码 Agent 构建普及，但须纳入 **3.3.2 Platform 治理**，防止影子 Agent 泛滥。

##### 3.3.5.6 一年展望：应用落地优先级（2027）

建议企业优先顺序：**高频可评测流程 > 知识密集型岗位 > 跨系统长流程 > 探索型创意场景**；与 **V 层回归成本** 和 **G 层风险** 匹配，避免“先上最难场景”导致规模化受阻。

#### 建议结尾

AI DC 的最终价值通过应用生态体现；**Agent 入口** 连接用户目标与企业业务系统。未来一年，应用生态成败的关键是 **能否在统一 Platform 上复制**，而非单个爆款 Demo。

---

### 3.3.6 生态横向治理：Harness 的 G / O 层与可信运行

#### 本节主旨

收束 Agent、数据、模型、应用共同依赖的治理；对应 ETCLOVG 的 **G**、**O**（及 **V**），强调 **Token / Watt** 治理与 **3.2 / 3.1** 衔接。

#### 关键观点

- 规模化瓶颈常在 harness 控制面，而非单点模型效果。
- Observability traces 须捕获身份与权限状态。
- 人工交接应包含意图、权限、预算、风险与未决项。
- **一年展望**：Agent **FinOps + SecOps** 合并治理；监管与内控将更多追问 **“谁授权 Agent 做了什么”**，而非仅“模型是否合规”。

#### 建议展开段落

##### 3.3.6.1 身份、权限与访问控制（G）

Agent 身份、委托关系、工具权限 **随任务生命周期变更**；一年内须支持 **即时吊销与会话冻结**。

##### 3.3.6.2 观测、审计与追溯（O）

Trace 须覆盖 **模型调用、检索片段、工具入参 / 出参、人工确认点**；支持 **按业务单据追溯**（如工单号、合同号）。

##### 3.3.6.3 验证体系与 trace-native 评估（V）

生产失败轨迹自动进入 **回归集**；版本发布门禁 = 模型变更 + Harness 变更 + 工具变更 **联合评测**。

##### 3.3.6.4 成本、Token、Watt 与 harness 权衡（衔接 3.1、3.2）

建立 **组织 / 部门 / 任务类型** 三级 Token 预算；同步观测 **Watt 峰值与 Token 峰值** 相关性，优化检查分层策略。

##### 3.3.6.5 安全、合规、人工确认与可信 AI（G）

重点防护 **工具层 prompt injection、越权调用、恶意工具供应链**；高风险操作 **双人复核或法务规则引擎** 接入。

##### 3.3.6.6 一年展望：治理成熟度模型（2027）

| 级别 | 特征 |
| ---- | ---- |
| L1 | 有 Gateway、有日志，无任务回放 |
| L2 | 有工作手册、有 Token 账单、有抽样评测 |
| L3 | Trace-native 回归、细粒度权限、标准人工交接 |
| L4 | 跨 Agent 编排可审计、FinOps 闭环、动态简化 Harness |

企业一年内目标建议定位于 **L2 → L3**，而非跳过基础直达“全自主”。

#### 建议结尾

**G + O + V** 平台化程度，决定 **Agent 入口** 能否在可追责前提下进入关键业务系统；未来一年，**治理投入应前置**，与 Platform 同步建设，否则 Agent 扩面将遭遇合规与成本双重反弹。

---

### 3.3.7 本节小结

#### 建议总结要点

- **3T 框架**：**3.1 Watt** 物理承载 · **3.2 Token** 算力成本 · **3.3 Agent** 生态与 Harness。
- **3.3.1**：以 Agent 为入口；**统一工作台 + 多形态共存**。
- **3.3.2**：Agent 时代底座 = 3T + ETCLOVG；**Platform 化与 O/G 补强** 是一年建设重心。
- **3.3.3**：AI-Ready Data **管道化、带 SLA、记忆分级**。
- **3.3.4**：**模型服务化 + Router 分层** 成为主流运营方式。
- **3.3.5**：应用从建议走向 **可审计半自动执行**。
- **3.3.6**：**L2→L3 治理成熟度** 是一年内务实目标。

#### 一年行动建议（给企业架构师，提纲级）

1. **收敛入口**：规划企业 Agent 工作台，冻结零散 Copilot 集成。  
2. **建设 Platform**：优先 durable workspace、沙箱池、Tool Registry、trace 与 Token 细账。  
3. **封装工具**：legacy 系统 Agent-ready 化与协议标准并行。  
4. **数据 SLA**：知识刷新与记忆治理纳入 Agent 上线门禁。  
5. **模型服务分层**：Gateway + Router + 任务级评测，而非单模型扩容。  
6. **治理前置**：工作手册、权限、人工交接与回归集与试点同步，而非上线后补丁。

#### 建议核心表述

企业构建 Agentic AI DC，须 **以 Agent-centric 牵引 Application 层**，并以 **Token-centric Infra** 与 **Watt-centric Facility** 协同支撑。站在 2026 看 2027，胜负手不再是“是否拥有大模型”，而是 **能否在统一 Agent Platform 上，以可观测、可验证、可担责的方式规模化运营 Agent**。
